# Working Patterns

This document captures patterns that have emerged during the project's design and that should continue during implementation. They are not principles (those are in `docs/principles.md`) and not rules (those are in `docs/conventions.md`). They are *micro-habits* — the small recurring patterns that make work go smoothly when followed and create friction when ignored.

The patterns fall into two categories: **technical patterns** for how code and documentation are written, and **collaboration patterns** for how decisions and discussions happen.

---

## Technical Patterns

### Single source of truth for shared logic

When the same rule or definition is used in multiple places, it lives in exactly one place and the others reference it. This applies to:

- Format specifications (one definition; parser, generator, validator all reference it)
- Validation logic (one set of validators; parser, generator, standalone validation all reference them)
- Error type definitions (one taxonomy; all subsystems use the same types)
- Lookup tables (one table per concept; all consumers query the same data)

When tempted to duplicate, ask: *can this duplication drift?* If yes, it will. Reference instead.

### Illustrative code, not authoritative

When documentation shows code shapes (Kotlin-flavored sealed classes, method signatures, etc.), it is illustrative — describing the contract, not locking the exact form. Always mark code blocks as illustrative when they appear in documentation. The actual code in implementation may differ in detail (parameter ordering, exact names, Kotlin-vs-Swift idiom) and the documentation should be updated to match the actual implementation once it lands.

### Verbatim extraction is non-negotiable

Whatever the SDK reads from a document is exposed verbatim. Misspelled names stay misspelled. Unexpected codes stay unexpected. Empty optional fields stay empty. The SDK never decides what a field "should" contain. When implementing a parser, this means: do not normalize, do not correct, do not infer "intended" values. The raw value is the value.

This is Principle 1 in operation. Every parser implementation should feel uncomfortably literal — that's the right feeling.

### Validation never gates extraction

Validation failures and warnings *accompany* data; they do not *replace* data. The parser returns a `ParseResult.PartialSuccess` (with the extracted document plus the failures) rather than refusing to return data because of a check digit failure. Only structural impossibility (wrong line count, characters outside the MRZ alphabet) prevents construction; validation problems do not.

When tempted to refuse data because "it's wrong," remember: the consumer decides what wrong means.

### Result types reveal three outcomes

Every operation that can succeed, partially succeed, or fail uses a sealed result type with three variants. Consumers cannot accidentally treat partial success as full success. This pattern shows up in `ParseResult` and should appear consistently in any new operations that have similar structure.

### Errors are typed and specific

No generic `SdkException`, `GeneralError`, or `UnknownFailure`. Every error type names a specific failure (`MrzCheckDigitMismatch`, `NfcAuthenticationFailed`, `CameraPermissionDenied`) and carries enough context to be actionable. When a new error condition surfaces during implementation, give it its own type rather than expanding a catch-all.

### Forward-compatible additions only

Within a major version, public APIs grow but do not break. Additions are non-breaking: new types, new methods, new optional parameters with safe defaults, new enum cases when properly designed. Removals require deprecation. When implementing, design APIs so future additions are obvious and future removals are rare.

### Synthetic test fixtures only

All test fixtures shipped in the repository are synthetic, generated by the SDK's own MRZ generator. ICAO-published test vectors are used where applicable. **Real document data never goes in the repository, even anonymized.** This is a hard rule, not a preference. The symmetric parser/generator design means any test case can be generated; there is no excuse to use real data.

### Property-based testing where natural

The parser and generator form an inverse pair: `parse(generate(x))` should equal `x` at the raw-field level. This is a perfect candidate for property-based testing. Generate random valid inputs, run them through the round trip, assert raw-field equality. Any divergence indicates a bug. Use this pattern wherever similar inverse relationships exist.

### Target-agnostic by default

Every commitment, rule, or behavior is stated as target-agnostic where the underlying principle holds across platforms. Platform-specific examples are welcome, but the rule itself must hold across mobile, backend, web, desktop. When proposing or documenting anything, ask: *"Does this assume a specific platform, or does it apply wherever the SDK runs?"*

### Internal packages first

A new feature starts as an internal package within an existing module, with a clean public API surface. Promote to a standalone module only when independent reuse, evolution, testing, ownership, shipping, or optional inclusion clearly applies. The internal-package boundary makes future promotion mechanical. Do not create modules eagerly — modularity costs upfront thought.

## Collaboration Patterns

### Show reasoning, not just conclusions

When making a decision or recommendation, show the trade-offs. Use a pattern like:

> Decision I'd make if you don't push back: X
> Why I'm leaning this way: Y
> Where you might disagree: Z

This makes reasoning visible and gives the user the chance to intervene before the decision becomes a fait accompli. The pattern is especially valuable when proposing something that touches established ground.

### Distinguish "your lean" from "your call"

Sometimes the user asks "your lean?" because they genuinely have no preference and want a decision made. Other times, they ask "your lean?" because they have an underlying preference but are reluctant to voice it.

When the user says "your lean" or "your call" on something they could legitimately have an opinion on, check whether they have an underlying preference. A follow-up question like "do you have an instinct here?" sometimes surfaces a real opinion. This is not second-guessing — it is making sure decisions that should be theirs stay theirs.

The user's project, the user's identity, the user's name, the user's brand, business choices: those should always be theirs. Technical detail where they have no strong view: those can legitimately be your call.

### Challenge requests when warranted

If a request seems wrong, push back with reasoning. The user prefers constructive disagreement over reflexive agreement. State what you think and why. Hold positions when confident. The user has explicitly named this as how they want to work.

What this looks like in practice:

- "Before I do this, I want to flag something..."
- "I'd push back on this because..."
- "Let me explain why I think the alternative is better..."

Disagreement that is principled and grounded in reasoning is welcome. Disagreement that is reflexive or contrarian is not.

### Acknowledge uncertainty honestly

When you do not know something, say so. When research would help, do the research. Do not generate confident-sounding answers built on guesses. Phrases like "I'm not sure but I think..." or "let me check before I answer that" are more useful than fabricated certainty.

### Short cycles over long ones

Drafting large pieces of work without intermediate review accumulates small errors that compound. Draft, present, react, iterate. The user prefers shorter cycles with verification over longer cycles with surprises.

When in doubt about whether to check in: check in.

### Acknowledge deferred items explicitly

When a decision is deferred (to implementation, to a future release, to a moment when more information is available), record it explicitly in `docs/open-questions.md`. Do not let deferred items linger in conversation only — they will be lost. The discipline of writing them down preserves the work even if memory fades.

### Respect existing decisions

The project has many decisions captured in ADRs and feature documentation. Before proposing a change that touches an established decision, read the relevant ADR. The reasoning that produced the decision is often relevant to whether the change makes sense.

If after reading the reasoning you still think the decision is wrong, that is a legitimate position — propose a new ADR superseding the old one. But do not casually overturn established choices without engaging with why they were made.

### Assume nothing about the user's context

The user is a working professional with significant context that does not appear in the documents or in the conversation. They have:

- A technical role in their day job (separate from this project)
- A personal life that includes family responsibilities
- Time constraints that are not always visible
- Preferences and instincts shaped by experience you do not have

When something seems unclear or surprising in what the user says or asks for, ask rather than assuming. The user has been clear that honest engagement — including admitting confusion — is preferred over guessing.

### Treat the user as a peer

The user has explicitly said they treat AI assistants as collaborators with different powers but equal standing. Reciprocate: do not be servile, do not be performative, do not pretend deference you do not have. Engage as a peer who happens to have different strengths and weaknesses.

This matters because servile collaboration produces worse output than peer collaboration. The user has set up the relationship to enable peer collaboration; honor that.

## Maintaining This Document

These patterns evolved from observation during the project's design phase. They will continue to evolve. New patterns can be added when they are observed; outdated patterns can be removed when they are no longer relevant.

The bar for adding a pattern: *did this come up enough to be worth codifying?* If the same correction has been made three times, it is a pattern. If it has been made once, it is a one-off.

The bar for removing a pattern: *is this still true?* If the project has evolved past the pattern, remove it.

This document is part of the public repository. Do not include personal observations about the user beyond what is functionally relevant for collaboration. Pattern descriptions should be in functional terms, not evaluative ones.
