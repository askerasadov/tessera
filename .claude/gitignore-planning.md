# `.gitignore` Planning

This document describes what should and should not be committed to the public repository for this project. It is planning material for AI assistants (and humans) making decisions about what to add, what to exclude, and what to keep entirely outside version control.

A proper `.gitignore` file will be generated when implementation begins. This document captures the reasoning so the `.gitignore` can be derived from it consistently.

---

## What Goes Into the Public Repository

### Source code

- All Kotlin source files in `commonMain`, `androidMain`, `iosMain`, etc.
- Android-specific code in `androidMain` (Activities, Composables, etc.)
- iOS-specific Swift wrappers and SwiftUI components
- Any platform-specific implementations of cross-platform interfaces

### Build configuration

- `build.gradle.kts`, `settings.gradle.kts`, version catalogs
- Gradle wrapper files (the `gradle-wrapper.jar` and `gradle-wrapper.properties`)
- Any module-level build files
- iOS project files (`.xcodeproj`, `Package.swift` for SPM, `.podspec` for CocoaPods)
- KMP configuration

### Documentation

- The entire `docs/` folder (principles, architecture, scope, conventions, versioning, features, decisions, etc.)
- The entire `.claude/` folder (working patterns, known pitfalls, this file, session handoff template)
- The project root `README.md`
- The project root `CLAUDE.md`
- License file (`LICENSE`)
- Contributor documentation (when written)

### Test fixtures

- Synthetic test fixtures (generated by the project's own MRZ generator)
- ICAO-published test vectors used in tests
- Test configurations and setup files

### CI/CD configuration

- GitHub Actions workflows (`.github/workflows/`) when GitHub is the chosen platform
- Linting and formatting configurations
- Code coverage configurations

### Project metadata

- `.gitattributes` for line ending normalization
- `.editorconfig` for consistent code style across editors
- `CHANGELOG.md` (Keep a Changelog format)

## What Does NOT Go Into the Public Repository

### Anything that names specific organizations

The project is vendor-neutral. No specific organization is named in any committed file — not as customer, partner, anchor user, or in any other capacity. This rule has been violated zero times so far in the design phase, and it must continue. If a stray reference appears in a commit, it is a bug to fix immediately.

This rule applies to:

- Source code comments
- Documentation
- Test fixtures
- Commit messages
- Pull request descriptions
- Issue templates and content
- Any other text in the repository

### Real document data

No real MRZ data, no real chip data, no real document images, no real personal information. Even if the data is public (e.g., a passport image found online), it does not go in the repository.

The synthetic-fixtures-only rule is absolute. The symmetric parser/generator design means any test case can be generated; there is no excuse to use real data.

### Personal information

- Personal phone numbers, email addresses, physical addresses
- The user's name as project owner is acceptable in license files; beyond that, attribution is minimal
- No biographical information about the project's author or contributors beyond what is conventional for open source attribution

### Internal context and notes

- Notes from conversations with specific stakeholders
- Internal discussions about commercial relationships, business strategy, or organizational politics
- Drafts of communications to specific parties
- Internal design conversations that include private context

If working notes accumulate during development, they live in a separate folder outside the repository (e.g., `~/projects/[project-name]-private/notes/`). They are never committed.

### Credentials and secrets

The standard list:

- API keys, tokens, passwords
- Private keys, certificates with private material
- Database connection strings
- Internal service URLs that should not be public
- Configuration files with embedded secrets

A pre-commit hook or CI check should catch any of these. Even with such a check, vigilance during commits is the first line of defense.

### Temporary working files

- IDE-specific files (`.idea/` for IntelliJ except for shared run configurations, `.vscode/` for VS Code)
- Operating system files (`.DS_Store`, `Thumbs.db`)
- Build outputs (`build/`, `target/`, `out/`, `Pods/`, `DerivedData/`)
- Temporary backup files (editor-specific)
- Local environment files (`local.properties`, `.env.local`)
- **Working notes in dot-prefix directories** — three categories of AI-session-driven working content that live in hidden folders, each gitignored as a whole directory: session handoffs (`.handoffs/SESSION-HANDOFF-*.md`, per-session end-of-work records), code/doc alignment recaps (`.recaps/RECAP-*.md`, periodic audits checking docs still match code), and spec conformance notes (`.conformance/CONFORMANCE-NOTES-*.md`, per-release audits verifying SDK behavior against external specifications). The handoff template lives in `.claude/session-handoff-template.md` and is committed; the directories themselves and their dated working notes are not.
- **External specification reference copies** (`.spec/`) — local copies of external technical specifications (primarily ICAO Doc 9303 PDFs and text-extracted equivalents) consulted during implementation. Never committed: redistribution of external specs may be restricted by their licensing terms. The project's "cite-and-implement" approach references section numbers in code/docs, never verbatim spec content.

## What Goes Into a Separate Private Repo (or Stays Off Version Control)

For the user's own working notes during development:

- Conversation transcripts that include private context (such as the design conversation that produced these documents)
- Internal commentary on stakeholder decisions
- Personal reminders, todos, or non-public roadmap considerations
- Drafts of communications to specific parties

These can live in a sibling private repository (e.g., `[project-name]-private`) or simply outside version control entirely. They never appear in the public repository.

---

## What `CLAUDE.md` and `.claude/` Should Contain

`CLAUDE.md` and the `.claude/` folder are public — they go into the open-source repository. They contain:

- Public-facing guidance for AI assistants working on the project
- Patterns and pitfalls described in functional, non-personal terms
- Document maps and orientation material
- General working conventions

They do *not* contain:

- Personal evaluations of specific contributors (including the user)
- Information about the user's identity beyond what is in license files
- Internal context, business strategy, or stakeholder notes
- Anything that would be embarrassing or harmful to the user if read by a stranger

If guidance about the user's specific working style is useful for AI assistants, it is phrased as functional patterns ("when the user says X, check whether Y") rather than personal characterization ("the user tends to Z").

## Handling Mistakes

If a private piece of information is committed to the repository accidentally:

1. **Stop immediately.** Do not push the commit if it has not been pushed.
2. **If already pushed:** the information is now public on the internet, even after deletion. Treat it as exposed. Rotate any credentials, remove the information, and force-push if the project is small enough that history rewriting is acceptable.
3. **Add a check** to prevent recurrence. A pre-commit hook, a CI scan, or a manual review step.
4. **Do not panic.** Mistakes happen. The discipline of catching them early matters more than perfection.

## Maintaining This Document

This document grows when new categories of decisions surface. As implementation begins, real `.gitignore` entries should be derivable from this content. If a `.gitignore` rule is added without corresponding reasoning here, the reasoning should be added so future contributors understand the rule.

When this document and the actual `.gitignore` disagree, the source of truth is determined by which one is correct for the project's current state. If the disagreement is genuine, both should be updated to align.
