## [0.1.0] - 2026-05-19

The first internal release. Pure parsing, generation, and validation for all five ICAO Doc 9303 MRZ formats (TD1, TD2, TD3, MRV-A, MRV-B), the error taxonomy, the lookup-table machinery, transliteration profiles (ICAO default + AZE country-specific), and the pluggable telemetry interface (contract-only at this release; first emitters in 0.2.0 and 0.6.0). JVM target enabled; additional targets activate per-release as the corresponding reading methods land. Includes a pre-tag spec-conformance verification pass against ICAO Doc 9303 (findings F1–F25, recorded across PRs [#43](https://github.com/askerasadov/Tessera/pull/43)–[#48](https://github.com/askerasadov/Tessera/pull/48)) and a follow-up 2026-05-19 empirical pass that expanded the AZE transliteration profile to its full systematic phonetic Anglicization pattern (8 overrides) plus a doc-drift cleanup from a deep alignment audit. 577 tests.

Per [ADR-007](docs/decisions/0007-strict-backward-compat-from-0x.md), the public API committed in this release is held to strict backward compatibility across the `0.x` line.

### Changed

- `AzeTransliterationProfile` expanded from a single schwa override to a full **systematic phonetic Anglicization** override set per the 2026-05-19 pre-tag empirical pass. Seven new override pairs (each applied to upper and lower case): `Ç/ç → CH`, `Ğ/ğ → GH`, `Ş/ş → SH`, `X/x → KH`, `C/c → J`, `J/j → ZH`, `Q/q → G`. The original `Ə/ə → A` schwa override is unchanged. Four of the new overrides (`C`, `J`, `Q`, `X`) target characters already in the MRZ alphabet, which required the profile to consult its override map **before** the MRZ-alphabet passthrough check — implementation detail captured in the profile's KDoc and in [ADR-009](docs/decisions/0009-transliteration-profiles.md) "Implementation Note: Override Lookup Order". `IcaoDefaultTransliterationProfile` retains its original lookup order (its conservative defaults never override an MRZ-alphabet letter). Primary citation: the ALA-LC romanization table for the AZE Latin alphabet (US Library of Congress / British Library bibliographic standard); empirical corroboration: sample AZE-issued documents + a fluent speaker's worked examples. The two overrides not explicitly in ALA-LC (`J → ZH` and `C → J`) are tracked separately in `docs/open-questions.md` under "AZE profile `J → ZH` and `C → J` empirical basis"
- [ADR-009](docs/decisions/0009-transliteration-profiles.md) Context section reframed (a second time, after the 2026-05-18 conformance reframe). Previously characterized the AZE divergence as a single load-bearing schwa edge case. Now describes two distinct patterns: (1) characters outside Annex G's coverage (the original schwa case, BGN/PCGN + ICAO no-expansion chain) and (2) characters Annex G covers but the issuing state's phonetic conventions diverge from (`Ç → CH`, `Ğ → GH`, `Ş → SH`) plus characters already in the MRZ alphabet that get Anglicized digraph encoding (`X → KH`, `C → J`, `J → ZH`, `Q → G`). Both patterns derive from the ALA-LC romanization table + the diacritic-stripping step the MRZ alphabet requires. The ALA-LC chain (ALA-LC romanize → strip diacritics for MRZ) generalizes the existing BGN/PCGN+ICAO schwa chain rather than replacing it

Coverage now includes British nationality classes (GBD/GBN/GBO/GBP/GBS), Kosovo (RKS), the European Union (EUE), United Nations documents (UNO/UNA/UNK), other international organizations (XPO/INTERPOL, XES/OECS, XMP/PAM, XOM/Order of Malta, XDC/SADC), stateless and refugee codes (XXA/XXB/XXC/XXX), the deprecated codes ANT (Netherlands Antilles) and NTZ (Neutral Zone) per Part F for backward compatibility on still-circulating documents, the synthetic UTO ("Utopia") specimen code per Part G (categorized as OTHER — not a real country), and ICAO's IAO code per Part H. ~272 total entries. Categorized per `CountryCodeCategory` (STATE / ORGANIZATION / STATELESS / REFUGEE / HISTORICAL / OTHER). Pre-tag conformance-verification finding (F7); cite-and-implement approach per the "External spec data licensing strategy" framing. ISO codes verified against published ISO 3166/MA listing; §5 extensions read from `spec/9303_p3_cons_en.txt`
- `IcaoDefaultTransliterationProfile` (via `buildIcaoLatinMappings`) now covers every codepoint in ICAO Doc 9303 Part 3 §6.A (Annex G). The previous starter set was already comprehensive against Latin-1 Supplement (U+00C0-00DE) and most of Latin Extended-A (U+0100-017D); the 2026-05-18 pre-tag conformance audit (F5) identified three remaining gaps that are now closed: `Ŋ` (U+014A "Eng") → `N`, `ŋ` (U+014B) → `N`, and `ẞ` (U+1E9E, capital sharp s) → `SS` (the lowercase `ß` was already handled). The full Annex G Latin coverage is now in place for both uppercase and lowercase variants. Where Annex G permits multiple recommended transliterations (Ä → "AE or A", Å → "AA or A", Ñ → "N or NXX", Ö → "OE or O", Ü → "UE or UXX or U"), the no-expansion variant is chosen consistently — documented in the profile's KDoc
- `domain/errors/MrzSexCharacterX` — new warning type emitted by `MrzValidator` when the MRZ sex field contains the character `X`. Per the 2026-05-18 pre-tag conformance audit (F16), the ICAO Doc 9303 2021 Eighth Edition (Part 4 §4.2.2.2 and the equivalent sections in Parts 5/6/7) lists the canonical MRZ sex characters as `M`, `F`, `<` only — `X` is reserved for the VIZ per the spec's Note p/f clauses. Real-world practice has nonetheless adopted `X` in the MRZ for non-binary or unspecified-sex documents. The SDK now accepts `X` (Principle 1 — reader, not oracle: do not refuse documents the real world actually issues) and surfaces this warning so consumers can identify the spec deviation and decide how to handle it. Matches the existing `MrzPersonalNumberCheckDigitFiller` pattern for documented real-world deviations
- `DocumentTypeCodeTable` populated with the full `P`-prefix harmonized set per ICAO Doc 9303 Part 4 §4.4 ("Document Codes"): `PE` (Emergency passport), `PO` (Official/service passport), `PR` (Refugee passport), `PT` (Alien/Non-citizen passport), `PL` (Laissez-passer passport), `PM` (Military passport). Part 4 §4.4 mandates the harmonized set for new MRPs from 1 January 2026 (already in effect at tag) and for all MRPs from 1 January 2028. Pre-tag conformance-verification finding (F15)
- `DocumentTypeCodeTable.AC` — TD1 Crew Member Certificate code per ICAO Doc 9303 Part 5 Appendix B. Categorized as `DocumentCategory.OTHER` since it does not fit any of the more specific categories. Pre-tag conformance-verification finding (F17)
- `domain/errors/MrzGenerationNumericInNameField` — new generation error type emitted by the primitive-input methods of `MrzGenerator` when the `primaryIdentifier` or `secondaryIdentifier` argument contains numeric characters. Per ICAO Doc 9303 Part 3 §4.6: "Numeric characters shall not be used in the name fields of the MRZ." The SDK does not strip digits silently (Principle 1 — reader, not oracle); the error carries the consumer's original input and the list of numeric characters encountered so the consumer can decide how to handle them. Pre-tag conformance-verification finding (F4)
- `CheckDigitTest` extended with the five canonical specimens from ICAO Doc 9303 Part 3 Appendix A: Example 1 (date `520727 → 3`), Example 2 (document number `AB2134<<< → 5`), Example 3 (TD3 composite `→ 8`), Example 4 (TD1 composite `→ 2`), Example 5 (TD2 composite `→ 8`). These are independent of the Part 4 specimens already in the suite and verify both the per-field check digit and the composite-input concatenation logic against canonical spec output. Pre-tag conformance-verification finding (F8)
- `MrzDateInferenceTest` extended with explicit coverage of filler-character date components (`<<` in any of the year / month / day slots, including all-filler). Per ICAO Doc 9303 Part 3 §4.8: "If all or part of the date of birth is unknown, the relevant character positions shall be completed with filler characters (`<`)." The parser's existing `toIntOrNull`-based component parsing already handles this correctly (filler components route to `RAW_ONLY` with `componentsFormCalendarDate = null`, the same path as any non-digit input). The new tests lock that behavior explicitly so the spec-named case is in the suite. Pre-tag conformance-verification finding (F9)
- [ADR-015](docs/decisions/0015-telemetry-contract-only-at-0-1-0.md) — Telemetry interface ships as contract-only at `0.1.0` with an open event hierarchy. Decision: lock the public surface of the pluggable telemetry interface in `0.1.0` per the `scope.md` commitment, but acknowledge that no `0.1.0` SDK module emits events (the dependency graph in `architecture.md` shows `telemetry` as depended on by `mrz-camera-{platform}` and `emrtd-nfc-{platform}` only — the first ships in `0.2.0`, the second in `0.6.0`). `TelemetryEvent` is an open interface rather than a sealed hierarchy because adding sealed subtypes post-`0.1.0` would be a breaking change under [ADR-007](docs/decisions/0007-strict-backward-compat-from-0x.md), and consumers route events to their observability backend rather than pattern-matching on them. The "nothing happens in 0.1.0" property is explicitly documented in the public KDoc on `TelemetrySink`, in `docs/features/telemetry.md`, and in this CHANGELOG entry (Principle 4 — honest about what we know). Index in `docs/decisions/README.md` updated
- New `docs/features/telemetry.md` feature document. Covers the `0.1.0` status (contract only, no events emitted), the public surface, the open-interface event-hierarchy rationale, the sink registration model, the redaction utilities, the boundary with the `logging` module, and the forward plan for when concrete events ship (camera in `0.2.0`, NFC in `0.6.0`)
- `telemetry` module: pluggable telemetry interface (per [ADR-015](docs/decisions/0015-telemetry-contract-only-at-0-1-0.md), [`docs/features/telemetry.md`](docs/features/telemetry.md))
  - `TelemetrySink` interface — single `record(event: TelemetryEvent)` method. Consumers implement to route events to their own observability stack. The KDoc explicitly notes that no `0.1.0` SDK module emits events; registering a sink in `0.1.0` has no observable effect until camera (`0.2.0`) or NFC (`0.6.0`) modules ship
  - `TelemetryEvent` open interface — single `name: String` property. Open (not sealed) because future event types are added in the releases of the modules that emit them, and additive sealed subtypes would break exhaustive matching under ADR-007. Diverges intentionally from the project's sealed result-type pattern (`ParseResult`, `TransliterationResult`); rationale in ADR-015
  - `NoOpTelemetrySink` object — the default sink; discards every event. Registered as the initial value in the sink registry so SDK modules can always look up a sink without a null check
  - `TelemetrySinkRegistry` singleton object — `register(sink)`, `reset()`, `current`. Matches the `TransliterationProfileRegistry` precedent (mutable singleton, "register at startup before concurrent use" contract). Once SDK modules begin emitting events, the registered sink may be called from any thread; consumer implementations must be thread-safe (documented but not enforced in `0.1.0`)
  - `Redaction.redactMrzLine(line)` utility — masks ASCII letters and digits with `'*'`, preserves filler `'<'` and overall length. The only redaction helper shipped in `0.1.0`; more helpers (document number, name, date-of-birth) ship alongside the events that need them. Designing speculative helpers now would lock the wrong surface under ADR-007

### Removed

- The `TestInfrastructureSmokeTest` placeholder in `mrz-core` (replaced by real tests as implementation slices landed)
- Empty `.gitkeep` placeholder files in source directories that now contain real Kotlin files
- `MrzGenerationMissingRequiredField` error type — removed pre-tag during the 2026-05-19 full alignment audit (Phase D, test-coverage discipline). The type was defined as a public sealed subtype of `MrzGenerationError` but had no production path: every required parameter on the primitive-input generator methods is non-nullable in Kotlin, and the data-model-input methods take fully-constructed `MrzDocument` instances whose `CommonFields` are all non-nullable too. Kotlin's type system enforces required-field presence at compile time, so no runtime code path could emit this error, and no test produced it. Under [ADR-007](docs/decisions/0007-strict-backward-compat-from-0x.md)'s strict-backcompat rule, keeping the type would have locked it into the API forever; removing pre-tag is the cheap correction. If a future generator API takes Map-like input where required fields can be omitted at runtime, the type can be re-added then
