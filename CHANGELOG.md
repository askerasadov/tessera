# Changelog

All notable changes to this project are documented here.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html). Within the 0.x line, the project follows strict backward compatibility per [ADR-007](docs/decisions/0007-strict-backward-compat-from-0x.md): MINOR additions are non-breaking, and breaking changes (when they occur) bump MAJOR.

## [Unreleased]

### Added

- `.claude/rules/types-module-discipline.md` — new path-scoped rule (`paths: "types/**"`, auto-loaded for AI assistants whenever working in the `types/` module). Codifies the discipline that emerged during the `domain` → `types` rename in PR [#78](https://github.com/askerasadov/tessera/pull/78) and ADR-016's "Follow-up Cleanup" reasoning: the `types` module holds types only (classes, interfaces, enums, type aliases, companion objects) and nothing else — no top-level functions, no top-level non-const properties, no extension functions on third-party types, no top-level `object` declarations used as function namespaces. When shared non-type code is needed, the convention is to create a new module (e.g., `tessera-utils`) rather than relaxing the discipline; broadly-named modules like `common`, `core`, or `utils` accumulate unrelated helpers over years and become junk drawers. The rule preserves the meaning the deliberately-chosen `tessera-types` artifactId promises before it locks under [ADR-007](docs/decisions/0007-strict-backward-compat-from-0x.md) at first Maven Central publish (0.1.1, per [ADR-016](docs/decisions/0016-maven-coordinates-and-first-publish.md)). Mirrors the two-surface placement precedent of the `folder-organization` rule: AI-facing rule file in `.claude/rules/` (path-scoped, auto-loaded), human-facing summary in `docs/conventions.md` (new "Module Boundaries" section). Cross-references between the two name each other as siblings
- [ADR-016](docs/decisions/0016-maven-coordinates-and-first-publish.md) — Maven Central coordinates, lockstep versioning, and first publication at 0.1.1. Locks the published groupId as `io.lightine.tessera` (aligned with the already-committed package root and backed by the verified Sonatype namespace from PR [#75](https://github.com/askerasadov/tessera/pull/75)); the artifactId convention as `tessera-<module>` (Firebase/Spring/Ktor-style prefix for self-identification across dependency tools, security scans, and IDE views); versioning as lockstep (all modules share one version, bumped together at each release, matching the project's coordinated-release roadmap shape); a `tessera-bom` artifact alongside the modules for version alignment in user `dependencies { }` blocks; and the first Maven Central publication at version 0.1.1 (the `v0.1.0` tag remains the internal milestone; `v0.1.1` is the first publicly-distributed version, lands with the publishing infrastructure as normal work). First-publish scope is all five current modules (`tessera-mrz-core`, `tessera-domain`, `tessera-emrtd-core`, `tessera-telemetry`, `tessera-logging`) plus `tessera-bom`; no snapshot builds at 0.x. Verified during pre-decision analysis that the `tessera-<module>` pattern scales cleanly across every planned future module in `docs/scope.md` (camera at 0.2.0, image reading at 0.3.0, UI at 0.5.0, NFC stack at 0.6.0, plus the Beyond-1.0 capabilities including chip signature verification, mDoc, selfie capture, liveness, face matching, VC holder, access-controlled session). All decisions lock under [ADR-007](docs/decisions/0007-strict-backward-compat-from-0x.md) once first publish ships. iOS distribution channel (CocoaPods vs Swift Package Manager) remains separately deferred per the "Distribution channels" entry in `docs/open-questions.md`, tied to iOS target activation per `docs/scope.md`. The `domain` module name is flagged as a follow-up rename cleanup — `domain` is somewhat generic; renaming to something more semantic (e.g., `mrz-types`, `tessera-types`) is preferable before 0.1.1 ships to avoid locking the name under ADR-007. Publishing infrastructure work (Gradle `maven-publish` plugin wiring, PGP signing key, Dokka for Javadoc, POM metadata, BOM module scaffolding, Sonatype credentials, publishing workflow) is a separate slice that coordinates with the 0.1.1 release work. A small follow-up doc PR will update the `docs/open-questions.md` "Distribution channels" entry to point at this ADR once both this ADR and PR #75 have merged
- `.claude/agents/doc-consistency-reviewer.md` — first project-level subagent. Read-only reviewer that enforces the "Documentation Sync (When You Change X, Update Y)" rules from `CLAUDE.md` mechanically: given a diff or PR, it identifies what changed, cross-references against the seven sync rules (public-API → feature doc, significant decision → ADR, error type → taxonomy + test, etc.), and reports which docs should have been touched but weren't. Constrained tool set (Read / Grep / Glob / Bash — no Edit / Write), sonnet model, manual invocation only via the `Agent` tool with `subagent_type: "doc-consistency-reviewer"`. Output format: compliant rules / missing updates / ambiguous cases / one-line summary. Does NOT make subjective doc-quality judgments, run tests, or expand scope into general code/security review. CLAUDE.md "Maintaining the Documentation System" placement table extended with a new row covering subagents (the fifth AI-facing-infrastructure home, alongside CLAUDE.md / rules / skills / hooks). Replaces the implicit "Claude reviews its own work" pattern with a delegated review in an isolated context window — no main-conversation pollution, mechanical rule-checking, easier for the user to invoke deliberately ("have the doc-consistency-reviewer check this") rather than relying on me remembering to self-review
- `scripts/private-content-scan.sh` — automated scanner for private-content terms in tracked files. Wired into a new PreToolUse hook in `.claude/settings.json` that fires on every `Bash(git push *)` call Claude makes, blocking the push (exit code 2) if the script finds anything that isn't a documented false positive. Manual pushes (from a regular terminal) run the script directly. The script reads scan terms from the gitignored `.claude/private-content-terms.local` (one substring per line, per-contributor); the terms themselves are the private context being protected, so the file is gitignored by design. False-positive allowlist (extended regex matching `InvalidData`, `"Azerbaijan"` as ISO 3166-1 data, and `azerbaij.pdf` URL filenames in ALA-LC citation links) is hardcoded in the script. `.gitignore` updated to exclude the terms file. `CLAUDE.md` "Run the private-content scan" rule rewritten to reference the automated hook + the manual script invocation; `.claude/git-workflow.md` section 4 expanded with the new first-time-setup steps and the exit-code contract. Replaces the prior manual rule that depended on me remembering to run the grep — now mechanical, per Anthropic's "use hooks for actions that must happen every time, zero exceptions" guidance
- `SECURITY.md` at project root — private vulnerability disclosure policy (GitHub Security tab + email fallback), supported-versions matrix, response timeline targets, in-scope / out-of-scope guidance. Added as part of the pre-public-sharing readiness pass after the v0.1.0 release tag landed (the repo went public during 0.1.0 work; this PR adds the protection scaffolding before the repo is shared with contributors)
- `CONTRIBUTING.md` at project root — short pointer file directing contributors to `docs/conventions.md`, `.claude/git-workflow.md`, `docs/versioning.md`, `docs/testing.md`, `docs/principles.md`, `docs/open-questions.md`, the PR template, and `SECURITY.md`. Resolves the "CONTRIBUTING.md at project root" entry in `docs/open-questions.md`
- `.github/CODEOWNERS` — default code owner mapping for the repo (`@askerasadov` for all paths). Paired with the planned `main` branch protection rule "Require review from Code Owners," only the listed owners' approvals satisfy the required-approval gate
- `.github/dependabot.yml` — monthly Gradle dependency updates and monthly GitHub Actions version updates with `chore(deps)` commit-message prefix and `dependencies` labels. Security updates are enabled separately via the GitHub UI and route through these same `package-ecosystem` entries
- `.github/workflows/check.yml` — GitHub Actions CI workflow running `./gradlew check` plus `bash scripts/check-cross-references.sh` on every pull request to `main` and every push to `main`. JDK 21 / Temurin, Gradle wrapper, read-only cache on PRs. Exists so the planned branch-protection rule "Require status checks to pass before merging" has something meaningful to wait on
- `docs/contributor-setup.md` — per-platform (macOS / Linux / Windows) walkthrough for one-time machine setup: cloning, configuring Git identity, generating an SSH signing key, registering it on GitHub as a Signing Key, and optionally setting up `gpg.ssh.allowedSignersFile` for local signature verification. Cross-linked from `CONTRIBUTING.md` as the "First-time setup" entry point. The macOS path is the project-verified one as of 2026-05; Linux and Windows follow the same logic but have not been end-to-end verified against branch protection
- `scripts/check-markdown-links.sh` — new doc-quality validator complementing `scripts/check-cross-references.sh`. Catches clickable inline markdown links in committed docs whose targets are not visible to a fresh cloner — links that point to gitignored or deleted files and silently 404 on GitHub. Uses `git ls-files` as ground truth rather than the local filesystem, so references to gitignored files that happen to exist on the maintainer's machine are correctly flagged. Directory targets are treated as resolved if the directory contains tracked files. Bash (~120 lines), no new dependencies, macOS bash 3.2 compatible. Wired into `.github/workflows/check.yml` as a new `Run markdown link validator` step after the existing cross-reference validator. Each script catches a distinct class of defect: this one for clickable links, the cross-reference one for backtick-only prose mentions

- `docs/contributor-setup.md` macOS step 3e + Section 4 troubleshooting — corrected a mental-model inaccuracy in the post-reboot SSH-signing guidance shipped in PR [#71](https://github.com/askerasadov/tessera/pull/71). ## [0.1.0] - 2026-05-19

The first internal release. Pure parsing, generation, and validation for all five ICAO Doc 9303 MRZ formats (TD1, TD2, TD3, MRV-A, MRV-B), the error taxonomy, the lookup-table machinery, transliteration profiles (ICAO default + AZE country-specific), and the pluggable telemetry interface (contract-only at this release; first emitters in 0.2.0 and 0.6.0). JVM target enabled; additional targets activate per-release as the corresponding reading methods land. Includes a pre-tag spec-conformance verification pass against ICAO Doc 9303 (findings F1–F25, recorded across PRs [#43](https://github.com/askerasadov/Tessera/pull/43)–[#48](https://github.com/askerasadov/Tessera/pull/48)) and a follow-up 2026-05-19 empirical pass that expanded the AZE transliteration profile to its full systematic phonetic Anglicization pattern (8 overrides) plus a doc-drift cleanup from a deep alignment audit. 577 tests.

Per [ADR-007](docs/decisions/0007-strict-backward-compat-from-0x.md), the public API committed in this release is held to strict backward compatibility across the `0.x` line.

- `telemetry` module: pluggable telemetry interface (per [ADR-015](docs/decisions/0015-telemetry-contract-only-at-0-1-0.md), [`docs/features/telemetry.md`](docs/features/telemetry.md))
  - `TelemetrySink` interface — single `record(event: TelemetryEvent)` method. Consumers implement to route events to their own observability stack. The KDoc explicitly notes that no `0.1.0` SDK module emits events; registering a sink in `0.1.0` has no observable effect until camera (`0.2.0`) or NFC (`0.6.0`) modules ship
  - `TelemetryEvent` open interface — single `name: String` property. Open (not sealed) because future event types are added in the releases of the modules that emit them, and additive sealed subtypes would break exhaustive matching under ADR-007. Diverges intentionally from the project's sealed result-type pattern (`ParseResult`, `TransliterationResult`); rationale in ADR-015
  - `NoOpTelemetrySink` object — the default sink; discards every event. Registered as the initial value in the sink registry so SDK modules can always look up a sink without a null check
  - `TelemetrySinkRegistry` singleton object — `register(sink)`, `reset()`, `current`. Matches the `TransliterationProfileRegistry` precedent (mutable singleton, "register at startup before concurrent use" contract). Once SDK modules begin emitting events, the registered sink may be called from any thread; consumer implementations must be thread-safe (documented but not enforced in `0.1.0`)
  - `Redaction.redactMrzLine(line)` utility — masks ASCII letters and digits with `'*'`, preserves filler `'<'` and overall length. The only redaction helper shipped in `0.1.0`; more helpers (document number, name, date-of-birth) ship alongside the events that need them. Designing speculative helpers now would lock the wrong surface under ADR-007
  - All types live in the `io.lightine.tessera.telemetry` package within the `telemetry` module

- [ADR-014](docs/decisions/0014-unicode-normalization-strategy.md) — Unicode normalization strategy for transliteration. Decision: normalize consumer input to NFC via platform-native normalizers (`java.text.Normalizer` on JVM/Android, `precomposedStringWithCanonicalMapping` on iOS, `String.prototype.normalize` on JS/Wasm) exposed through a Kotlin Multiplatform `expect`/`actual` declaration. For `0.1.0` only the JVM `actual` is implemented; other platforms supply their one-line `actual` when their targets activate. The post-normalization, pre-transliteration form is exposed to consumers per Principle 5. This is the implementation-strategy companion to ADR-009 and the output of the first formal invocation of the Pre-Release Tech-Stack Review rule (landed in [#31](https://github.com/askerasadov/tessera/pull/31)/[#32](https://github.com/askerasadov/tessera/pull/32) and applied to `0.1.0` in [#33](https://github.com/askerasadov/tessera/pull/33)). Cross-referenced from `docs/features/transliteration.md` (new "Unicode Normalization Before Profile Lookup" section) and from ADR-009's `Related Decisions`. Index in `docs/decisions/README.md` updated

- `mrz-core` module: transliteration profile API (per [ADR-009](docs/decisions/0009-transliteration-profiles.md), [`docs/features/transliteration.md`](docs/features/transliteration.md))
  - `TransliterationProfile` interface — `identifier: String` and `toMrzAlphabet(normalizedInput: String): TransliterationResult`; documented contract that input is assumed already-NFC (orchestration belongs above the profile, per ADR-014)
  - `TransliterationResult` sealed class with `Success(output: String)` and `Failure(unmappedCharacters: List<UnmappedCharacter>)` variants; profiles with a documented fallback policy return `Success`, profiles without one surface unmapped characters via `Failure` (Principle 7 — fail loudly, fail informatively)
  - `UnmappedCharacter` data class carrying the offending character and its zero-based position in the post-normalization input (lives in `domain.vocabulary` so it can be referenced by domain-side error types as well as by transliteration results)
  - `TransliterationProfileRegistry` object — `register` / `lookup(identifier)` / `all()`; ICAO default pre-registered; documented "register at startup before concurrent use" contract
  - `IcaoDefaultTransliterationProfile` — implementation of ICAO Doc 9303 Part 3 Section 6 (Annex G) recommendations under the no-expansion convention (`Ä → A`, `ü → U`) plus the explicit multi-character transliterations (`Æ → AE`, `Œ → OE`, `ß → SS`, `Þ → TH`, `Ð → D`, `Ĳ → IJ`, `Ə → E` per the schwa example called out in ADR-009). Fallback policy: any character not in the table maps to filler `<`, so this profile always returns `Success`. Ships as a deliberate starter set (tracked in `docs/open-questions.md` under "Transliteration profile coverage completeness")
  - `AzeTransliterationProfile` — the first country-specific profile required by ADR-009 ("at least one"), identified by the ISO 3166-1 alpha-3 code `AZE`. Mostly aligns with the ICAO default but overrides the Latin schwa pair (`Ə → A`, `ə → A`) per observed practice for this issuing state — the canonical example ADR-009 cites as the reason profile divergence exists in the SDK at all. Same fallback policy as ICAO (any unmapped character maps to filler `<`); pre-registered alongside the ICAO default. Per the project's vendor-neutral framing, the country is identified by its ISO code in code and never named in prose
- `mrz-core` module: generator integration with the transliteration subsystem
  - `MrzGenerator.generateTD3`, `generateTD2`, `generateTD1`, `generateMrvA`, `generateMrvB` — per-format primitive-input methods per the illustrative shape in `docs/features/mrz-generation.md`. - `mrz-core` module: date-in-calendar validation (TD3, both birth and expiry)
  - `MrzDate.componentsFormCalendarDate: Boolean?` tri-state signal added with a default of `null`; populated by `parseBirth` and `parseExpiry`. `null` when raw components did not parse as 2-digit numerics, `true` when components form a real `LocalDate` for at least one candidate century (covers successful inference and out-of-window calendar-valid dates), `false` when no candidate year forms a calendar date. The default keeps existing `MrzDate(...)` constructor call sites compiling unchanged (Principle 9). One nuance: like any `data class` property addition, `equals`/`hashCode`/`copy`/`componentN()` now include the new field — code that manually constructs an `MrzDate` with all other fields populated and compares it to a parser-produced instance will find them unequal where they would have been equal before (manual default = `null`, parser-set = `true`). No call site in the project's own code hits this path; flagged here for completeness
  - `MrzValidator.validateTD3` emits `MrzDateNotInCalendar` for `dateOfBirth` (position offset 13 on line 2) and `dateOfExpiry` (position offset 21) when `componentsFormCalendarDate == false`. Calendar-valid dates that the parser rejects via the inference window (e.g., expiry > 50 years out) do not produce this failure — the date IS in the calendar, just outside the heuristic
- `domain` module: birth-date warning taxonomy
  - `MrzBirthDateImplausiblyOld` warning carrying `rawYear`, `rawMonth`, `rawDay`, `referenceDate: LocalDate`, and `thresholdYears: Int`
- `mrz-core` module: birth-date warning (TD3 birth)
  - `MrzDate.componentsExceedBirthAgeLimit: Boolean?` tri-state signal added with a default of `null`; populated only by `parseBirth`. `true` when the parser falls to `RAW_ONLY` because every calendar-valid past candidate exceeds the parser's age cap (`MAX_PLAUSIBLE_AGE_YEARS = 130`); `false` when the parser succeeded or when no past calendar-valid candidate exists; `null` for `parseExpiry`-produced dates, direct-construction defaults, non-numeric components, or when no candidate forms a calendar date. Same data-class equality nuance as `componentsFormCalendarDate` — code that manually constructs an `MrzDate` with all other fields populated and compares it to a parser-produced instance will find them unequal where they would have been equal before
  - `MrzValidator.validateTD3` emits `MrzBirthDateImplausiblyOld` for `dateOfBirth` when `componentsExceedBirthAgeLimit == true`. Threshold reported on the warning is sourced from `MrzDate.MAX_PLAUSIBLE_AGE_YEARS` (visibility lifted from `private` to `internal` so the validator can read the same constant the parser uses for inference). Under current-era reference times (year ≤ ~2130) the cap is unreachable in practice; the warning matters for replay scenarios, audit pipelines, and far-future reference times that consumers may pass explicitly. Closes the documented commitment in `mrz-validation.md` "Date Range Conventions" that the validator surfaces an implausibly-old birth signal at a 130-year threshold

### Removed

- The `TestInfrastructureSmokeTest` placeholder in `mrz-core` (replaced by real tests as implementation slices landed)

- `MrzGenerationMissingRequiredField` error type — removed pre-tag during the 2026-05-19 full alignment audit (Phase D, test-coverage discipline). The type was defined as a public sealed subtype of `MrzGenerationError` but had no production path: every required parameter on the primitive-input generator methods is non-nullable in Kotlin, and the data-model-input methods take fully-constructed `MrzDocument` instances whose `CommonFields` are all non-nullable too. Kotlin's type system enforces required-field presence at compile time, so no runtime code path could emit this error, and no test produced it. Under [ADR-007](docs/decisions/0007-strict-backward-compat-from-0x.md)'s strict-backcompat rule, keeping the type would have locked it into the API forever; removing pre-tag is the cheap correction. If a future generator API takes Map-like input where required fields can be omitted at runtime, the type can be re-added then
